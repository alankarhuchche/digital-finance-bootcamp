# Visual Experience Guide

## Product feel

The site should feel like a premium interactive banking-practitioner reference.

It should not feel like:
- a flat blog,
- a generic course,
- a wiki,
- a crypto brochure,
- a static slide deck,
- an AI-generated text dump.

Every major topic should combine:
- clear practitioner writing,
- structured comparison,
- visual explanation,
- concise judgement,
- useful interaction where it genuinely helps.

## Visual principles

Use visuals to explain trade-offs, systems and decisions.

Prefer:
- comparison matrices for model choices,
- flow diagrams for processes,
- timelines for regulatory or market evolution,
- scale bars for relative size,
- callouts for practitioner judgement,
- maps only when geography genuinely matters,
- scatter plots only when a two-axis trade-off is meaningful,
- lightweight animations where they clarify movement, sequencing or state.

Avoid:
- decorative charts,
- visuals that repeat the paragraph above them,
- dense tables with long prose in every cell,
- diagrams that look like generic consulting slides,
- chart junk,
- unexplained icons,
- noisy animations,
- toy-like effects.

## Premium quality bar

A page is not complete just because it has text.

For important topics, aim for at least one of:
- a comparison matrix,
- a flow diagram,
- a decision lens,
- a practitioner callout,
- a scale or timeline visual,
- a compact quiz that tests the key distinction.

High-value pages should usually have two or more structured elements.

Examples:
- Stablecoin Market Structure should compare issuer models and show reserve economics.
- DLT Basics should compare platforms and show how to choose based on use case.
- Privacy should show visibility, disclosure and confidentiality trade-offs.
- AML/Sanctions should show control points across the payment or token flow.
- Regulation should show who regulates what, not only describe the rules.

## Interaction rules

Interaction should clarify a decision or system.

Good interactions:
- switch between rails,
- compare two options,
- reveal stages of a process,
- show finality gaps,
- filter or group a complex map,
- explain a term on demand.

Poor interactions:
- animation only for decoration,
- hidden information with no reason,
- excessive hover-only behaviour,
- interactions that do not work on mobile,
- interactions that make the page harder to read.

## Animation rules

Animations should be restrained and useful.

Use animation for:
- showing movement through a rail or flow,
- staged reveal of finality or settlement,
- highlighting a selected comparison,
- making diagrams feel alive without distracting from the text.

Avoid:
- constant motion,
- bouncing elements,
- flashy gradients,
- carousel-like behaviour,
- animation that hurts readability,
- animation that ignores reduced-motion preferences.

Always respect reduced-motion settings where applicable.

## Existing visual components

Use existing components before creating new ones.

Current useful blocks include:
- `matrix`
- `comparison`
- `flow`
- `timeline`
- `map`
- `scale`
- `scatter`
- `callout`
- `case`
- `stack`
- `money-cards`
- `quiz`

Do not create a new block type unless:
- existing blocks cannot express the idea clearly,
- the new component will be reused,
- the visual adds real understanding,
- mobile behaviour is planned,
- reduced-motion and accessibility are considered.

## Diagram quality rules

A diagram should have one clear job.

Before adding a diagram, define:
- what question it answers,
- what decision it clarifies,
- what the reader should notice first,
- what the reader should remember.

Keep diagram labels:
- short,
- specific,
- banking-oriented,
- free of hype,
- readable on mobile.

Avoid long prose inside diagram boxes. Put explanation below or beside the diagram.

## Matrix quality rules

Matrices should compare decision-relevant attributes.

Good columns:
- liability,
- redemption,
- finality,
- privacy,
- operating model,
- governance,
- regulatory treatment,
- bank relevance,
- watch-outs.

Poor columns:
- generic benefits,
- vague pros and cons,
- marketing claims,
- unsupported performance numbers,
- "innovation potential".

Cells should be concise. If every cell needs a paragraph, the matrix is too dense.

## Page layout rules

Each important page should have rhythm.

Good rhythm:
1. opening thesis,
2. structured explanation,
3. visual comparison,
4. practitioner implication,
5. caveats,
6. short quiz or recap.

Avoid:
- long uninterrupted text,
- stacked tables with no breathing space,
- too many callouts in a row,
- repeated headings with the same shape,
- pages that feel like pasted research notes.

## Mobile and accessibility

Every visual must remain usable on mobile.

Check:
- no title overlap,
- no clipped labels,
- scroll affordance where needed,
- tap targets are usable,
- text remains readable,
- diagrams do not rely only on hover,
- keyboard focus remains visible,
- reduced-motion preferences are respected.

If a visual cannot work on mobile, simplify it.

## Visual credibility

Do not make visuals look more certain than the facts allow.

For market sizing:
- include dates,
- distinguish supply from transaction volume,
- distinguish trading volume from payment usage,
- avoid unsupported precision,
- caveat estimates.

For platform comparisons:
- avoid unsupported TPS claims,
- avoid "best" language,
- compare by use case and operating constraints.

For regulation:
- include jurisdiction and date where relevant,
- avoid implying proposed rules are final,
- avoid one-size-fits-all global claims.

## When adding or changing a topic

Before finishing, check:

- Does the page have enough visual structure for its importance?
- Does each visual explain something the text alone would not?
- Are diagrams and matrices concise?
- Does the page still feel premium?
- Does mobile still work?
- Are animations restrained and useful?
- Does the page avoid looking like an AI-generated article?

## Final review question

Before reporting completion, answer:

> Would this page still look credible if shown to a senior banking technology, payments, risk or strategy leader?

If the answer is no, improve the visual structure before finishing.
